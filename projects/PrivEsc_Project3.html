Inside attacker Privilage-Escalation

Purpose: In this project I demonstrate how a single misconfigured SUID binary can allow escalation from a restricted user account to full root compromise. My aim was to simulate a realistic penetration-testing scenario where careful enumeration of privilege boundaries exposed an anomalous binary, and to show how that weakness could be exploited and then remediated.

Methodology: I started by configuring a fresh Ubuntu virtual machine where I created a non-privileged user account to act as the attacker’s foothold. To simulate a misconfiguration commonly seen in practice, I introduced a root-owned binary into /usr/local/bin and applied the SUID bit. I began by confirming that my attacker account had no elevated privileges. I then enumerated the system to locate binaries with the SUID permission set. I then identified the unusual entry, inspected its properties, and executed it to test whether it could be abused. After confirming escalation, I validated the impact by reading a file normally restricted to root, and finally remediated the misconfiguration by removing the SUID bit and proving the escalation path no longer worked.

Process:

I began as the restricted attacker user. Running whoami and id confirmed I had no administrative privileges, and attempts to read /etc/shadow failed with “Permission denied.”

Next, I enumerated SUID binaries using find / -perm -4000 -type f. Alongside the standard system entries, an unusual binary named suid_adm appeared in /usr/local/bin.

Inspection with ls -l showed the binary was owned by root and marked with the SUID bit (-rwsr-xr-x), making it a candidate for privilege escalation.

Executing suid_adm dropped me into a new shell. Repeating whoami and id confirmed my user context had changed to root.

With elevated privileges, I was able to access /etc/shadow, validating the severity of the misconfiguration.

Finally, I remediated the issue by removing the SUID bit with chmod u-s. Verifying with ls -l showed normal permissions (-rwxr-xr-x), and re-executing the binary as the restricted user no longer provided elevated privileges.

Result:

Through this workflow I escalated privileges from an unprivileged account to root by exploiting a single misconfigured binary. I then demonstrated that removing the SUID bit immediately closed the escalation path.

Value:

This project highlights how attackers approach privilege escalation in practice. From my perspective as the attacker, the key steps were enumeration, anomaly detection, exploitation and validation. From the defender’s perspective, the project reinforces the importance of regularly auditing binaries with elevated permissions and removing unnecessary SUID bits. Including this case study in my portfolio demonstrates not only that I can identify and exploit misconfigurations, but also that I understand how to remediate them. It complements my other work on TLS and vulnerability scanning by adding a system-level exploitation example, reflecting scenarios that regularly appear in real penetration tests.
