Reconnaissance & Vulnerability Scanning

Objective

Demonstrate how attackers enumerate services, probe for weaknesses, and brute-force hidden content, while also showing how such scanning appears from the defender’s perspective. The project highlights both offensive discovery and defensive detection.

Environment & Threat Model

  • Target: self-configured Nginx webserver with HTTPS and a self-signed TLS certificate.
  • Server hosted a simple login page that logged inputs to a PHP file.
  • Attacker machine used Nmap, Nikto, and Gobuster.
  • Traffic captured using tcpdump and analysed in Wireshark.
  • Assumption: attacker has network access to the target system.

Methodology & Process

The target system was an Nginx webserver I configured, hosting HTTPS with a self-signed TLS certificate. The webserver had a simple login page that logged inputs to a PHP file.

The vulnerability scanning workflow began with service discovery to confirm exposed ports, followed by TLS enumeration to determine supported versions and cipher suites. Next, Nikto scanned for misconfigurations and sensitive files, and Gobuster brute-forced hidden directories. Finally, Wireshark was used to observe scanning traffic from the server’s perspective.

Service Discovery: Nmap confirmed that the target hosted HTTP and HTTPS. An initial scan identified Nginx on ports 80 and 443.

TLS Enumeration: Using Nmap’s ssl-enum-ciphers script, the server was found to accept TLS 1.0 and 1.1. After reconfiguring Nginx, a second scan verified only TLS 1.2+ remained enabled.

Nikto Scan: Generated hundreds of GET requests, probing for common files and directories. Discovered the deliberately placed PHP login page.

Network Capture: To demonstrate defender visibility, I ran tcpdump on the webserver while re-running Nikto. Wireshark showed the resulting flood of requests.

Results

The assessment confirmed two exposed services: HTTP on port 80 and HTTPS on port 443. TLS scanning initially revealed legacy protocols (TLS 1.0 and 1.1), which were removed through reconfiguration. Nikto flagged misconfigurations and discovered an exposed login form. Gobuster enumerated additional directories. Wireshark highlighted how noisy scanning traffic looks from the defender’s perspective.

Analysis

  • Demonstrated risk: Default server configs may allow insecure TLS versions and expose sensitive files.
  • Limitations: Scanning results often contain false positives; production environments may trigger alerts or rate-limiting.
  • Learning outcome: Gained practice chaining multiple reconnaissance tools and interpreting results from both attacker and defender views.

Defensive Relevance

  • System administrators should disable legacy TLS protocols and enforce modern cipher suites.
  • IDS/IPS can detect noisy scanner traffic like Nikto; defenders can tune rules to alert on such activity.
  • Validating scanner output prevents chasing false positives and reduces wasted remediation effort.

Ethical Context

All scanning was performed against a self-configured lab environment under my control. No external systems were targeted.
