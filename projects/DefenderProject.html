Objective

Demonstrate investigation and triage of a Microsoft Defender alert identifying repeated sign-in attempts from a suspicious external IP address targeting disabled accounts. The goal was to determine whether the activity represented malicious user activity (Initial Access / brute-force probing) or a benign authentication misconfiguration.

Environment & Threat Model

  • Platform: Microsoft Defender XDR / Microsoft Sentinel (Azure tenant lab environment).
  • Detection source: Built-in analytics rule "Sign-ins from IPs that attempt sign-ins to disabled accounts".
  • Alert severity: Medium (mapped to Initial Access / Persistence categories).
  • Observed behaviour: External IP generated multiple authentication attempts against disabled Azure AD accounts.
  • Error code observed: 50057 (User account is disabled).
  • Entity investigation performed on source IP address (ASN, geolocation, reputation score, correlated alerts).
  • Threat model assumption: activity may represent credential stuffing, brute-force enumeration, or reconnaissance of valid account names from hostile infrastructure.

Methodology & Investigation Process

  • Alert identified within the Microsoft Defender SOC queue and opened for triage.
  • Status set to In Progress and assigned to analyst to simulate Tier 1 ownership workflow.
  • Reviewed alert metadata, severity, rule description, and associated incident timeline.
  • Examined Attack Story and Incident Graph to understand entity relationships.
  • Pivoted to the related entity: source IP address.
  • Analysed IP enrichment data including ASN, geolocation, reputation score, and correlated alerts.
  • Assessed behavioural pattern for indicators of credential stuffing, brute-force enumeration, or reconnaissance activity.
  • Documented findings and applied final classification within the incident record.

The investigation began in the Microsoft Defender alert queue. A Medium severity alert titled “Sign-ins from IPs that attempt sign-ins to disabled accounts” was identified for triage and investigation. The alert category was mapped to Initial Access / Persistence. This stage represents standard Tier 1 monitoring workflow: identifying new alerts, reviewing severity, and determining priority.

After initial triage of the incident, the incident was classified as a 'false positive' (due to the its nature as a demo incident) and assigned to an agent.

The investigation pivoted to the source IP address associated with the authentication attempts.

    Investigation into the IP address revealed:
  • Geolocation: North Korea
  • ASN ownership
  • Suspicious reputation score
  • Multiple correlated alerts

    • IP enrichment is critical in assessing whether authentication attempts originate from hostile or anomalous infrastructure.

    Analysis

    • Observed behaviour: A single external IP address generated authentication attempts against one or more disabled accounts (error code 50057), consistent with account enumeration and credential testing patterns.
    • Risk: Repeated sign-in attempts from suspicious infrastructure can indicate password spraying / credential stuffing. If successful, this can lead to account takeover, persistence, and potential lateral movement depending on access granted.
    • Contextual indicators: IP enrichment showed high-risk context (geolocation, ASN ownership, reputation scoring) and correlated alert activity, increasing confidence that the activity was not routine user behaviour.
    • Limitations: This investigation was performed in a demo environment and raw sign-in telemetry in the Observed in organization view was not available for deeper validation (e.g., confirming successful sign-ins, Conditional Access outcomes, or session correlation).
    • Learning outcome: Practiced Tier 1 SOC workflow including alert triage, incident scoping, entity pivoting, enrichment analysis, classification, and documentation of findings.

    Defensive Relevance

    • Enforce strong authentication controls (e.g., MFA, Conditional Access) and block high-risk sign-ins where possible.
    • Review and tune detections for authentication anomalies (disabled account sign-in attempts, repeated failures from a single IP, and unusual geolocations).
    • Implement rate-limiting and lockout policies to reduce effectiveness of password spraying and credential stuffing attacks.
    • Use IP-based controls where appropriate (block known hostile IPs/ASNs) while recognising attackers can rotate infrastructure.
    • Ensure SOC playbooks include containment steps: user password reset, session revocation, and escalation when successful authentication is suspected.

    Ethical Context

    • This investigation was conducted using Microsoft-provided demo alerts in a controlled lab environment for educational and portfolio purposes.
    • No real user accounts, external systems, or production tenant resources were targeted or impacted.
