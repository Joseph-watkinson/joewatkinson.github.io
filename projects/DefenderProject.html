Microsoft Defender - Disabled Account Sign-In Investigation

Objective

Demonstrate investigation and triage of a Microsoft Defender alert identifying repeated sign-in attempts from a suspicious external IP address targeting disabled accounts. The goal was to determine whether the activity represented malicious user activity (Initial Access / brute-force probing) or a benign authentication misconfiguration.

Environment & Threat Model

  • Platform: Microsoft Defender XDR / Microsoft Sentinel
  • Detection method: Built-in analytics rule: "Sign-ins from IPs that attempt sign-ins to disabled accounts".
  • Alert severity: Medium
  • Observed behaviour: External IP generated multiple authentication attempts against disabled Azure AD accounts, whilst successfully signing into other active accounts.
  • Error code observed: 50057 (User account is disabled).
  • Entity investigation performed on source IP address (ASN, geolocation, reputation score, correlated alerts).
  • Threat model: activity may represent: credential stuffing or password spraying from a malicious origin

Methodology & Investigation Process

  • Alert identified within the Microsoft Defender queue and opened for triage.
  • Status set to In Progress and assigned to analyst to simulate Tier 1 workflow.
  • Reviewed alert metadata, severity, rule description and associated incident timeline.
  • Examined Attack Story and Incident Graph to understand alert relationships and any notable correlations.
  • Pivoted to investigating the source IP address due to nature of alert.
  • Analysed IP data including: ASN, geolocation, reputation score, and correlated alerts.
  • Assessed pattern for indicators of credential stuffing, brute-force or password spraying
  • Documented findings and applied final classification within the incident record.

The investigation began in the Microsoft Defender alert queue. A Medium severity alert titled “Sign-ins from IPs that attempt sign-ins to disabled accounts” was identified for triage and investigation. The alert category was mapped to Initial Access / Persistence. This stage represents standard Tier 1 monitoring workflow: identifying new alerts, reviewing severity, and determining priority.

The Incident Overview provided:

  • Detection rule logic.
  • Timeline of activity.
  • Incident severity.
  • Entity relationships via Attack Story.

Reviewing the incident context ensures the alert is not analysed in isolation and helps identify correlated activity.

After initial triage of the incident, the incident was classified as a 'false positive' (due to the its nature as a demo incident) and assigned to an agent.

The investigation pivoted to the source IP address associated with the authentication attempts due to the nature of the alert.

Investigation into the IP address revealed:

  • Geolocation: North Korea
  • ASN ownership
  • Suspicious reputation score
  • Multiple correlated alerts

Analysis

  • Observed behaviour: A single external IP address generated authentication attempts against one or more disabled accounts (error code 50057), whilst successfully signing into active accounts from the same IP address. Behaviour is consistent with password spraying or credential stuffing attacks.
  • Risk: Repeated sign-in attempts from suspicious infrastructure can indicate password spraying / credential stuffing. If successful, this can lead to account compromise and disruption through the disabling of innocent accounts.
  • Contextual indicators: IP data showed high-risk attributes (geolocation, ASN ownership, reputation scoring) and correlated alert activity, increasing likelihood that the activity was not routine user behaviour.
  • Next steps in investigation:If session data and successful sign-in data confounded suspicion, the alert would be escalated and the IP address likely blocked from attempting any further sign-ins.
  • Limitations: This investigation was performed in a demo environment and raw sign-in telemetry in the 'Observed in organization' tab was not available for deeper validation (e.g. confirming successful sign-ins or session correlation).
  • Learning outcome: Practiced Tier 1 SOC workflow including alert triage, incident scoping, entity pivoting, enrichment analysis, classification, and documentation of findings.

Defensive Relevance

  • Enforce strong authentication controls via MFA, conditional access and block high-risk sign-ins where possible.
  • Review and tune detections for authentication anomalies (disabled account sign-in attempts, repeated failures from a single IP, suspicious locations).
  • Implement rate-limiting and lockout policies to reduce the effectiveness of password spraying and credential stuffing attacks.
  • Use IP-based controls where appropriate to block known hostile IPs/ASNs..

Ethical Context

  • This investigation was conducted using Microsoft-provided demo alerts in a controlled lab environment for educational and portfolio purposes.
  • No real user accounts, external systems, or production tenant resources were targeted or impacted.
