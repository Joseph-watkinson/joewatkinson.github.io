TLS 1.2 RSA Key Exchange Decryption

Objective

Demonstrate how TLS 1.2 sessions configured with RSA key exchange are vulnerable if a server’s private key is compromised. Because RSA does not provide forward secrecy, previously recorded traffic can be decrypted retroactively.

Environment & Threat Model

  • Three VMs: client, attacker, and nginx webserver with a self-signed certificate.
  • Webserver restricted to TLS 1.2 RSA cipher suites only.
  • Client submitted login credentials over HTTPS.
  • Attacker captured traffic with tcpdump.
  • Threat model assumption: attacker has access to the server’s private key.

Methodology

The attacker captured the TLS handshake and application traffic into a .pcap file. Wireshark was used to confirm the cipher suite and locate the encrypted POST request.

The server’s private key was imported into Wireshark, enabling decryption of the TLS session.

Results

Wireshark successfully decrypted the captured HTTPS session. The plaintext POST request revealed the submitted username and password.

Analysis

  • Demonstrated risk: RSA key exchange exposes all past sessions if the private key is compromised.
  • Limitations: This applies only to RSA cipher suites; modern servers prefer ECDHE/DHE with forward secrecy.
  • Learning outcome: Ability to configure lab environments, capture/analyze traffic, and apply cryptographic knowledge in practice.

Defensive Relevance

  • Disable RSA-based cipher suites in TLS 1.2 and enforce forward-secret ciphers (ECDHE).
  • Regularly audit server configurations for weak ciphers and compliance.
  • Incident responders must consider: if a private key leaks, all previously captured traffic is vulnerable.

Ethical Context

This project was performed entirely in a self-controlled lab environment for educational purposes. No external systems were targeted.
