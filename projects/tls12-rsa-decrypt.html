TLS 1.2 RSA Key Exchange Decryption

Objective

Show how TLS 1.2 sessions configured with RSA key exchange are vulnerable when a server’s private key is exposed. RSA lacks forward secrecy, meaning an attacker who records traffic can later decrypt it retroactively.

Environment / Threat Model

  • Three VMs: client, attacker, nginx webserver (self-signed certificate).
  • Webserver restricted to RSA cipher suites under TLS 1.2.
  • Client submits login credentials over HTTPS.
  • Attacker captures traffic with tcpdump.
  • Assumption: attacker has obtained server private key.

Methodology

The attacker captured the TLS handshake and application data, saved into a .pcap. In Wireshark, filters confirmed RSA key exchange.

The server’s private key was imported into Wireshark, enabling decryption of the TLS session.

Results

Once the key was loaded, Wireshark successfully decrypted the HTTPS session. The plaintext POST request revealed submitted credentials.

Analysis

  • Demonstrated risk: RSA key exchange exposes all past sessions if the private key is compromised.
  • Limitations: Modern servers usually prefer ECDHE/DHE, which provide forward secrecy. RSA key exchange is mostly legacy.
  • Learning outcome: Ability to configure lab environments, capture/analyze traffic, and apply cryptographic understanding to practical exploitation.

Defensive Relevance

  • Organizations should disable RSA-based cipher suites in TLS 1.2 and enforce ECDHE.
  • Monitor server configs for weak cipher use; include this in compliance checks.
  • Incident responders should be aware: if a key is compromised, all previously recorded traffic is at risk.

Ethical Context

This exercise was performed entirely within a self-controlled lab environment for educational purposes.
